What "real transparency" actually requires

Right now, ledger.py writes trust events to a local JSON file — nobody outside your server can see it. Real transparency means writing those same events to a public blockchain, where anyone can look up the transaction and see the exact amount, timestamp, and campaign — permanently, with no way for anyone (including you) to edit or delete it afterward.

lightbulb
This is a two-part job: (1) a small Solidity smart contract deployed on a chain, and (2) rewriting ledger.py's record_event() to call it via web3.py instead of writing to a file. Everything else in the app — routes, templates, the trust score — stays exactly the same. Nothing gets built until you're ready — this document is just the plan.

Blockchain, in plain English

Five ideas explain almost everything you need to follow the rest of this guide.

menu_bookBlockchain

Your personal keychain for that notebook. It proves "this entry was written by me" — no username or password, just a secret key only you hold.

A practice copy of the notebook, using fake play money you get for free. Perfect for learning and testing before anything real is on the line.

Attestation ledger vs. real on-chain escrow

This is the biggest decision — it decides how much changes for donors, and how much risk is involved. For learning and for a first working version, the left option is the easier one to understand and build.

Easier to understand — start here

Attestation ledger

Like getting a stamped, tamper-proof receipt for something that already happened off-chain.

Donors still pay by bKash/Nagad/bank exactly like today. The blockchain just permanently records proof that each step happened — not the money itself.

  • No donor wallets needed — nothing changes for them
  • Cheap: a few cents (or less) of gas per admin approval
  • Matches the app's current design almost exactly
  • Small, easy-to-read contract — good for a first project
Bigger, harder project

Real on-chain escrow

Like putting money in a vending machine that only opens when the rules say so.

Donors send actual crypto to the smart contract itself, which holds and releases the money automatically — code enforces the escrow, not just a database flag.

  • Donors need a crypto wallet — a real change for them
  • Real money at risk — should be security-reviewed first
  • More Solidity concepts to learn (transfers, reentrancy, etc.)
  • A natural "phase two" once phase one is working
hub Second decision

Ethereum vs. Polygon — and why it barely matters yet

Good instinct bringing up Ethereum — it's the original smart-contract blockchain, and every network below "speaks the same language" it invented. The code you write, the wallet you use (MetaMask), and the programming language (Solidity) are identical across all of them. The only thing that changes between networks is one setting: which address you connect to.

school
So nothing is "wasted" by practicing on a cheaper network first — you're learning real Ethereum-family skills the whole time. Moving your finished contract to Ethereum mainnet later is a one-line change, not a rewrite.

The smart contract (attestation model)

A short, readable Solidity contract with one function per trust event. Each function just writes a permanent, public log entry — it never touches any money.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

contract TrustFundLedger {
  address public admin;

  event DonationRecorded(uint256 indexed campaignId, uint256 amount, address donor, string offchainTxId);
  event ProofRecorded(uint256 indexed campaignId, string proofHash);
  event WithdrawalRecorded(uint256 indexed campaignId, uint256 amount);

  modifier onlyAdmin() { require(msg.sender == admin, "Not authorized"); _; }
  constructor() { admin = msg.sender; }

  function recordDonation(...) external onlyAdmin { emit DonationRecorded(...); }
  function recordProof(...) external onlyAdmin { emit ProofRecorded(...); }
  function recordWithdrawal(...) external onlyAdmin { emit WithdrawalRecorded(...); }
}
visibility
Because these are public events, anyone can look up the full history of any campaign directly on the blockchain — with no dependency on your server staying online, or honest.

What changes in the Python app

Only one file changes when the time comes. Everything else — routes, templates, the trust score — is untouched.

# .env (stays on your computer only, never shared)
RPC_URL=https://eth-sepolia.g.alchemy.com/v2/<your-key>
PRIVATE_KEY=<your admin wallet's private key>
CONTRACT_ADDRESS=0x...

# ledger.py — record_event() gets a real body
from web3 import Web3
w3 = Web3(Web3.HTTPProvider(RPC_URL))
contract = w3.eth.contract(address=CONTRACT_ADDRESS, abi=ABI)

def record_event(event_type, payload):
  tx = contract.functions.recordDonation(...).build_transaction({...})
  signed = w3.eth.account.sign_transaction(tx, PRIVATE_KEY)
  tx_hash = w3.eth.send_raw_transaction(signed.raw_transaction)
  return tx_hash.hex()

What this costs to run, later

While testing, everything is free. Once live on a real network, each admin approval is one small transaction:

  • On Ethereum mainnet: roughly $2–30+ per approval, depending on network traffic
  • On Polygon mainnet: roughly $0.001–0.01 per approval
  • One-time contract deployment is a similar order of magnitude, paid once
  • The free tier of Alchemy/Infura comfortably covers a project like this

Glossary

RPC URL
The web address your app uses to "talk to" the blockchain network — like an API endpoint for the notebook.
ABI
A small file describing what functions a smart contract has, so other code knows how to call it.
Transaction hash (tx hash)
A unique ID for one entry written to the blockchain — like a receipt number anyone can look up.
Mainnet
The real, live network where transactions use real money.
Faucet
A free website that sends small amounts of test currency to a wallet address, for practicing on a testnet.
MetaMask
The most common wallet app — a browser extension that stores your keys and lets you approve transactions.
